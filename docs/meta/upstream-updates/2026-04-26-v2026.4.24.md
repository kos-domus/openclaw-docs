# OpenClaw 2026.4.24

Released 2026-04-25. Detected by Kos on 2026-04-26 with the local install still on **2026.4.15** — nine versions behind, and below the security floor set by 2026.4.23.

## Why this matters here

This entry exists to capture **two facts that affect day-to-day operations**:

1. **Security floor**: every version up to 2026.4.22 inclusive is below the patches landed in 2026.4.23. Treat anything older than 2026.4.23 as out-of-policy.
2. **Breaking change**: 2026.4.23 introduced a breaking change in the **plugin SDK / tool-result middleware** signature. Custom plugins compiled against 4.22 or earlier will not load on 4.23+ without rebuilding.

## Recommended action

| Path | When to choose |
|---|---|
| Upgrade directly to **2026.4.24** | Default. Get the security patches and the latest fixes in one move. Smoke test required (see below). |
| Stop at **2026.4.23** | Only if 4.24 introduces a regression you can't accept. 4.23 is the security floor. |
| Stay on 4.15 | Not acceptable past today (2026-04-26). Schedule the upgrade. |

## Smoke test before re-enabling traffic

After the install (`npm install -g openclaw@2026.4.24`) and gateway restart, run:

```bash
# 1. Provider auth round-trip
openclaw models list
openclaw models test --provider openai-codex
openclaw models test --provider gemini

# 2. Custom plugin / skill rebuild check
#    Plugins built against the old SDK signature will fail to register here.
openclaw plugins list --status
openclaw skills list --status

# 3. WhatsApp channel — affected by both the version jump AND the
#    in-flight reconnect-499 issue (see common-errors troubleshooting)
openclaw logs --channel whatsapp --follow &
# send a test message from another phone, expect inbound to land

# 4. Tool-result middleware — exercise an agent with a custom tool
openclaw run kos --skill openclaw-monitor --dry-run
```

Roll back if any of these fail; do not let agents auto-respond on 4.24 until the smoke test is green.

## Known regression in 4.24: bonjour mDNS advertiser crash

Confirmed during the 2026-04-26/27 rollout from 4.15 → 4.24:

```
[plugins] bonjour: restarting advertiser (service stuck in announcing for 17724ms ...)
[openclaw] Unhandled promise rejection: CIAO ANNOUNCEMENT CANCELLED
openclaw-gateway.service: Main process exited, code=exited, status=1/FAILURE
```

The watchdog introduced around 4.20 tries to re-advertise mDNS when the announcement state stalls. When the underlying `ciao` library cancels the prior announcement, the rejection is unhandled and crashes the gateway, which systemd then restart-loops. Stability bundle is written to `~/.openclaw/logs/stability/`.

**Workaround**: disable the `bonjour` plugin if your install does not rely on `*.local` mDNS discovery (most setups using Tailscale or Cloudflare tunnel don't):

```bash
openclaw plugins disable bonjour
systemctl --user restart openclaw-gateway.service
```

After the restart, gateway loads with 7 plugins instead of 8 and stays up.

> ⚠️ **Don't disable bonjour blindly** if local-network agents discover the gateway via `openclaw.local`. Check your channel configs first.

## Open issues not automatically solved by the version bump itself

These were live as of 2026-04-26, but 4.24 alone does not make them disappear unless the underlying cause is addressed:

- **Codex `refresh_token_reused`** — still requires per-agent login / token recovery, see [Common Errors](../../troubleshooting/common-errors.md#openai-codex-auth-failures).
- **Gemini saturation `All models failed`** — still requires a cross-provider fallback chain, see [Agent Fleet Reference](../../reference/agent-fleet-reference.md#cross-provider-fallback-rule-apr-25-2026).
- **WhatsApp `status=499` loops** are a split diagnosis now: if they stop once gateway uptime is stable, treat bonjour/systemd crash loops as the root cause; if they continue after gateway stability is confirmed, then fall back to the Baileys re-pair procedure in [Common Errors](../../troubleshooting/common-errors.md#whatsapp-reconnect-loop-status-499).
