# OpenClaw 2026.4.29

A large operator-facing release centered on message steering, people-aware memory, provider/runtime reliability, and another deep sweep across Telegram, WhatsApp, Slack, Discord, and startup behavior.

## Highlights

- Active-run messaging now defaults to **steer** semantics, with clearer queue behavior and visible-reply enforcement options.
- Memory grows into a people-aware wiki with aliases, provenance, relationship views, scoped recall filters, and bounded timeout fallbacks.
- NVIDIA joins the provider catalog, while Codex/OpenAI-compatible replay and streaming get safer behavior.
- Gateway diagnostics add startup timelines and more readiness visibility on slow or noisy hosts.
- Startup, plugin runtime dependencies, stale-session recovery, and channel reliability all got another serious hardening pass.

## Breaking or operator-impactful changes

- **Restrictive profiles no longer inherit `tools.exec` or `tools.fs` implicitly.** If an agent uses `minimal` or `messaging`, you now need explicit `alsoAllow` entries for those tools.
- `messages.visibleReplies` can now force visible user-facing output through the message tool path, which may expose assumptions in older channel workflows.
- More behavior keeps moving into plugin manifests and metadata snapshots, so old implicit runtime assumptions become less reliable over time.

## Fixes that matter here

- Telegram got stronger proxy, polling, webhook, timeout, quote-reply, send-bounding, and liveness fixes.
- WhatsApp got better outbound delivery truthfulness, liveness reporting, mention handling, reconnect recovery, and pairing-noise suppression.
- Gateway startup now tolerates more slow-host and plugin-runtime-dependency edge cases without looking dead or crash-looping.
- Stuck sessions, orphaned subagents, cron timeout cleanup, and Codex/OAuth edge cases all received direct fixes.

## Relevance for this docs repo

The local install is still on **2026.4.24**, so this repo is now behind four stable releases: **2026.4.25, 2026.4.26, 2026.4.27, and 2026.4.29**. There is no stable `2026.4.28` release tag to track.

The main pre-upgrade callout is the restrictive-profile change: any fleet config that relied on `tools.exec` or `tools.fs` being implicitly widened under `minimal` or `messaging` needs a smoke test before rollout.
