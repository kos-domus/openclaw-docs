## Objective

Process the 2026-04-26 ops report from Kos (Apr 25 daily wrap-up + Apr 26 upstream scan), translate the operational findings into OpenClaw documentation, and execute the operational fixes that are safe to launch without further user interaction.

## Context

- Local install was on **2026.4.15**. Upstream stable was **2026.4.24** since 2026-04-25 — nine versions behind, below the 2026.4.23 security floor.
- Three live ops issues were flagged by Kos:
  1. WhatsApp channel in reconnect loop with `status=499` every ~30 minutes; inbound at zero.
  2. OpenAI Codex auth failure (`refresh_token_reused` / `credential unavailable`).
  3. Gemini saturation: jobs that had only Gemini fallbacks exited with `All models failed`.
- Plugin compatibility check on the 4.15 install reported `none` — no custom plugins, so the 4.23 plugin-SDK breaking change was not a blocker for upgrading.

## Steps Taken

### 1. Update agent-fleet-reference.md to v3 with cross-provider fallback rule

Added a new "Cross-provider fallback rule" section to `docs/reference/agent-fleet-reference.md` codifying the lesson from the Apr 25 Gemini saturation: a fallback chain that stays inside one provider is not a fallback. Three-tier template (critical / standard / background), each with at least two distinct providers in the chain.

```yaml
# Before (insufficient)
primary: openai-codex/gpt-5.4
fallbacks: [gemini-2.5-flash]

# After (cross-provider)
primary: openai-codex/gpt-5.4
fallbacks: [gemini-2.5-flash, anthropic/claude-sonnet-4-6]
```

Also added an "OpenAI Codex auth fragility" subsection covering `refresh_token_reused` causes and per-agent login mitigations.

**Result**: doc bumped from v2 to v3 with new tags `fallback`, `providers`, sources extended with the Kos monitor report.

### 5. Diagnose the restart loop — bonjour mDNS regression

Pulled `journalctl --user -u openclaw-gateway.service` and found:

```
[plugins] bonjour: restarting advertiser (service stuck in announcing for 17724ms ...)
[openclaw] Unhandled promise rejection: CIAO ANNOUNCEMENT CANCELLED
openclaw-gateway.service: Main process exited, code=exited, status=1/FAILURE
openclaw-gateway.service: Scheduled restart job, restart counter is at 1.
```

The bonjour-watchdog (added around 4.20) re-advertises mDNS when announcement state stalls. When the underlying `ciao` library cancels the prior announcement, the rejection is unhandled and crashes the gateway. Stability bundles were being written to `~/.openclaw/logs/stability/` on every crash.

```bash
openclaw plugins disable bonjour
# Disabled plugin "bonjour". Restart the gateway to apply.

systemctl --user restart openclaw-gateway.service
```

**Result**: gateway came up clean with 7 plugins (was 8). PID stayed stable. Loopback `127.0.0.1:18789` answered HTTP 200 in 14ms. WhatsApp + Telegram providers initialized normally.

### 6. Verify WhatsApp recovered

After the gateway stabilized, the WhatsApp channel went from `499 reconnect loop` to `Listening for personal WhatsApp inbound messages` — without re-pairing the Baileys session. Hypothesis: the 499 loop reported by Kos was a downstream symptom of the bonjour-induced gateway restart loop, not a dead Baileys session. Each gateway restart was re-initialising the WhatsApp provider mid-handshake and the resulting reconnect attempts presented as 499s.

**Result**: WhatsApp recovered without manual intervention. Session re-pair was NOT needed.

## Key Discoveries

- **Fastify v5 sync preHandlers hang**: discovered earlier in the same SpesaBot work but worth elevating here — Fastify v5 hangs indefinitely on a sync preHandler that doesn't take a `done` callback. The pattern in older code (`function preHandler(req, reply): void`) silently fails: VTA-equivalent log shows the function ran to completion and set request state, but Fastify never proceeded to the route handler. Fix: declare the preHandler `async` (or accept `done` and call it). Symptom in the wild: only routes using `preHandler:` hang; routes that call the same auth function inline work fine. Worth adding to common-errors when a similar Claude Code/Node session shows up.
- **bonjour mDNS watchdog** introduced post-4.20 has an unhandled-rejection regression in 4.24 (`CIAO ANNOUNCEMENT CANCELLED`). On installs that don't actually need mDNS service discovery (Tailscale-served, Cloudflare-tunneled), disabling the plugin is the right answer rather than rolling back.
- **Symptoms can chain**: the user-visible symptom was a WhatsApp 499 reconnect loop. The root cause was a different plugin (bonjour) crashing the gateway, which in turn restarted WhatsApp mid-handshake. The lesson is to look at gateway uptime and crash logs *first* when channel-level symptoms show up at fixed intervals (here: ~30 minutes matched the systemd restart backoff, not a Baileys-side timeout).
- **Plugin-compatibility-none ≠ no-plugins-can-break**: the 4.23 release notes flagged a breaking change in the *custom* plugin SDK. Stock plugins (like bonjour) can still regress, and the breaking-change scan won't catch them.
